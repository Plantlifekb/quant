Quant architecture v1.0
Governed reference document — canonical, non‑drifting

1. Purpose and principles
Purpose:
Quant is a regime‑aware, multi‑model prediction system designed to generate tradable forecasts for financial instruments over multiple horizons, with full governance, auditability, and logical consistency.

Core principles:
- Regime awareness: Different models for different market regimes.
- Calibration: Confidence must reflect real‑world accuracy, not cosmetic scores.
- Governance: No silent changes, no schema drift, no ad‑hoc shortcuts.
- Verification: Every stage must be validated with real, complete outputs.
- Non‑drift: All future development must align with this document unless explicitly revised.

2. Data ingestion (5‑year governed history)
Objective:
Maintain a rolling 5‑year window of clean, normalized market data as the foundation for all modelling and regime work.

Inputs:
- Raw market data (prices, volumes, benchmarks, etc.) for all tickers of interest.

Responsibilities:
- Normalization:
  - Enforce a canonical schema with lowercase column names.
  - Enforce ISO‑8601 date formats.
- Continuity:
  - Ensure no missing dates for active tickers within the 5‑year window.
- Governance:
  - Store ingested data in governed locations with clear versioning and timestamps.

Canonical ingestion schema (Quant v1.0):
- date
- ticker
- company_name
- market_sector
- open
- high
- low
- close
- adj_close
- volume
- run_date

Rules:
- No derived fields.
- No portfolio‑level fields.
- No model‑dependent fields.
- Only static metadata may be added (and only with explicit versioning).

Outputs:
- A 5‑year, analysis‑ready dataset used by:
  - regime derivation
  - model training
  - performance evaluation

3. Regime derivation
Objective:
Classify each date (and optionally each ticker) into market regimes for trend and volatility, and determine tradability.

Inputs:
- 5‑year normalized market data.

Responsibilities:
- Compute raw regime indicators.
- Derive governed regime fields:
  - trend_regime
  - vol_regime
  - trend_regime_label
  - vol_regime_label
  - is_tradable_regime

Outputs:
- Regime‑annotated history.
- Merge‑ready regime file (e.g. regime_labels_v1.csv) keyed by date, ticker.

4. Regime‑specific models
Objective:
Train separate models for each regime (and horizon) using only data from that regime.

Inputs:
- Regime‑annotated 5‑year history.
- Feature sets for each ticker and horizon.

Responsibilities:
- Train one model per regime (and per horizon if applicable).
- Track performance per model:
  - accuracy
  - calibration
  - drawdowns
- Store models and metadata in governed locations.

Outputs:
- Regime‑specific model artifacts.
- Performance metadata per model and regime.

5. Daily prediction engine (regime‑aware)
Objective:
Generate daily predictions for each ticker using the correct regime‑specific models.

Inputs:
- Today’s features for each ticker.
- Today’s regime fields:
  - trend_regime
  - vol_regime
  - is_tradable_regime
- Regime‑specific models and metadata.

Responsibilities:
- Regime resolution.
- Model selection.
- Prediction generation.
- Confidence estimation (calibrated).

Outputs:
- Raw predictions per ticker and horizon, including:
  - date, ticker
  - pred_5d, pred_20d
  - trend_regime, vol_regime, is_tradable_regime
  - confidence metrics

6. Weekly retraining (Saturday)
Objective:
Keep models and regime classifier up to date with the latest 5‑year window and outcomes.

Schedule:
- Runs every Saturday.

Inputs:
- Updated 5‑year history.
- Realized outcomes.
- Latest regime annotations.

Responsibilities:
- Retrain regime classifier.
- Retrain all regime‑specific models.
- Recompute performance metrics and calibration.
- Archive old models and logs with timestamps.
- Update performance outputs.

Outputs:
- Refreshed models and regime classifier.
- Updated performance metrics and logs.

7. Gating logic
Objective:
Prevent nonsense or non‑tradable predictions from entering the governed outputs.

Inputs:
- Regime information.
- Volatility measures.
- Confidence scores.
- Model agreement metrics.
- Tradability rules.

Responsibilities:
- Determine is_tradable_regime and any additional tradability flags.
- Apply gating rules to raw predictions:
  - zero out or suppress predictions that fail rules.

Outputs:
- Gated predictions:
  - pred_5d_gated
  - pred_20d_gated

8. Output builders
Objective:
Produce fully populated, governed CSV outputs with no blank columns or partial data.

Inputs:
- Gated predictions.
- Realized outcomes.
- Regime labels.
- Performance metrics.

Responsibilities:
- Build the following governed outputs:
  - predictions_v1.csv
  - weekly_actuals_v1.csv
  - top5_master_5y_v1.csv
  - perf_latest_v1.csv

Rules:
- No half‑filled CSVs.
- No blank columns.
- No dropped functionality.
- All column names lowercase.
- All dates ISO‑8601.

Outputs:
- Fully populated, schema‑compliant CSVs ready for dashboards and analysis.

9. Schema validator
Objective:
Enforce schema integrity for all governed Quant v1.0 outputs.

Responsibilities:
- Validate:
  - column names
  - column order
  - presence of required fields
- Fail loudly and narratably on:
  - missing files
  - schema mismatches
  - reordering
  - silent corruption

Outputs:
- Pass/fail status with clear diagnostics.
- Non‑zero exit code on failure.

10. Confidence and logical consistency
Hard invariant:
- Confidence scores must be empirically calibrated.
- If a model outputs 0.99 confidence, it must be correct ~99% of the time over appropriate out‑of‑sample data.

Rules:
- No cosmetic confidence fields.
- Calibration must be periodically checked and logged.
- If calibration fails:
  - trigger retraining, recalibration, or gating adjustments.

11. Governance and development rules
1. Step‑by‑step development:
- Implement one module at a time.
- Do not proceed until outputs are real, verified, and narratable.

2. Full verification:
- No placeholder files.
- No blank columns.
- No partially implemented outputs.

3. Canonical architecture file:
- This document (Quant architecture v1.0) is the single source of truth.
- Any change must be explicitly written into a new version (e.g. v1.1).

4. No assumptions:
- If something is unclear or a file is needed, the system must ask.
- No guessing.

5. No drift:
- All new code, scripts, and outputs must align with this document.

🔒 Quant v1.0 — Script Header Standard (Mandatory for All Modules)

Every Python script inside C:\Quant\scripts\ must include a header that follows the same structure, tone, and governance conventions as ingestion_5years_quant_v1.py.

Header Requirements (Mandatory)
Each script must begin with a header containing:

1. Versioning Block
- Module name
- Quant version (v1.0)
- Purpose of the script
- High‑level description

2. Inputs
- Exact file paths or directories the script reads from

3. Outputs
- Exact file paths the script writes to
- Expected schema and column order (if applicable)

4. Governance Rules
- No schema drift
- No silent changes
- Lowercase column names only
- ISO‑8601 date formats only
- No writing outside designated directories
- Deterministic, reproducible behaviour

5. Logging Rules
- Must use logging_quant_v1.py
- Must log start, end, and key events
- Must log errors in governed format

6. Encoding Rules
- All CSV outputs must be UTF‑8
- All logs ASCII‑safe

7. Dependencies
- Required Python packages
- Internal Quant modules used

8. Provenance
- Script participates in governed Quant v1.0 pipeline
- All changes must be documented and versioned

This rule applies to:
- ingestion
- regimes
- features
- models
- predictions
- gating
- outputs
- validation

Every script must begin with a header that mirrors the structure and governance style of ingestion_5years_quant_v1.py.

This ensures:
- consistency
- auditability
- readability
- maintainability
- zero ambiguity
- zero drift

⭐ Quant v1.0 — Backtest & Attribution (Governed Performance Pipeline)
12. Realized Returns (Canonical Source of Truth)
Objective:
Provide a deterministic, governed dataset of realized returns used by all downstream modules.
Inputs:
- Governed ingestion prices (5‑year window).
Responsibilities:
- Compute close‑to‑close arithmetic returns.
- Enforce ISO‑8601 dates and lowercase columns.
- No cost adjustments, no leverage, no model‑dependent fields.
Output:
- quant_realized_returns_v1.csv
Rules:
- No schema drift.
- No hidden data sources.
- Deterministic: identical inputs must produce identical outputs.
- This file is the only valid source of realized returns for backtest and attribution

13. Governed Backtest Engine (PnL Generation)
Objective:
Produce a fully reproducible, auditable PnL series using governed inputs only.
Inputs:
- quant_realized_returns_v1.csv
- quant_portfolio_weights_tradable_v1_osqp.csv
- backtest_config_v1.json (leverage, cost model)
Responsibilities:
- Align weights and returns by date/ticker.
- Apply leverage scaling (if configured).
- Compute turnover and transaction costs.
- Compute daily and cumulative PnL.
- Compute gross and net exposure.
Output:
- quant_backtest_pnl_v1.csv
Rules:
- No external or legacy data.
- No ad‑hoc return definitions.
- No silent changes to cost or leverage logic.
- Deterministic and fully reproducible.

14. Governed Attribution Engine (Ticker, Sector, Daily)
Objective:
Explain PnL in a fully reconciled, auditable manner.
Inputs:
- quant_backtest_pnl_v1.csv
- quant_realized_returns_v1.csv
- quant_portfolio_weights_tradable_v1_osqp.csv
- ticker_reference.csv
Responsibilities:
- Compute raw per‑ticker contributions.
- Scale contributions to match governed PnL exactly.
- Aggregate to ticker, sector, and daily attribution.
Outputs:
- quant_attribution_ticker_v1.csv
- quant_attribution_sector_v1.csv
- quant_attribution_daily_v1.csv
Rules:
- Attribution must reconcile exactly to governed backtest PnL.
- No schema drift.
- No hidden scaling factors.
- Deterministic and reproducible.

15. Archiving of Legacy Artefacts
Objective:
Prevent non‑reproducible or pre‑v1.0 artefacts from contaminating governed outputs.
Archived File:
- quant_backtest_legacy_pre_v1.csv
(formerly quant_backtest.csv)
Reason:
- Produced before Quant v1.0 governance.
- Not reproducible from canonical inputs.
- Not aligned with governed cost/leverage rules.
- Excluded from attribution and performance analytics.
Rules:
- Legacy artefacts must never be used in governed modules.
- All governed modules must reference only v1.0 canonical inputs.

16. Performance Pipeline Invariants
Hard invariants:
- Realized returns must come from quant_realized_returns_v1.csv.
- Backtest PnL must come from quant_backtest_v1.py.
- Attribution must reconcile exactly to governed PnL.
- No module may compute its own returns.
- No module may read legacy artefacts.
- Any change requires a version bump (v1.1, v1.2, etc.).
